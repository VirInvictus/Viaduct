# Viaduct — AI Agent Architecture & Guidelines

This file is the definitive reference for AI agents (Claude, Gemini, etc.) and human contributors working on **viaduct**. Project-scoped rules here **override** anything in `~/CLAUDE.md` for work inside this repo.

Before you touch code, read `README.md`, `spec.md`, `roadmap.md`, and `patchnotes.md`. They carry invariants this file does not repeat.

---

## 1. What viaduct Is

**viaduct is a port of NetNewsWire to GTK4 and Rust. That is the entire project.**

NetNewsWire is open-source, battle-tested, and has already solved every hard problem in this domain — feed parsing, date parsing, conditional GETs, article deduplication, coalesced UI updates, memory discipline. We are translating that solution from Swift/AppKit to Rust/GTK4 so Linux users can have it. **We are not designing a new RSS reader.** We are not re-litigating its architectural choices. We port first, then we make it work on Linux.

If you catch yourself thinking "I have a better idea" — you don't. Go read the corresponding Swift file in `.netnewswire/` and port *that*. The app is local-first (local accounts plus Inoreader, and no other remote per §7.3), renders articles through exactly one locked-down `WebKitWebView` (§7.4), and holds to a strict memory budget. Targets **GNOME 50+** and **libadwaita 1.7+** on Wayland.

Current version: **v2.9.0**. Tagged stable points: v1.6.0 (collects the v1.5.x stability arc — workspace split, Meson, NNW-parity audit, visual identity, theme-contrast picker, adaptive-layout fixes, working YouTube playback via iframe wrapper). The v1.7.x series filled in the obvious missing UI (Add Feed dialog, right-click context menus on sidebar + timeline). v1.8.0 added sync-on-open + periodic-refresh preferences and the written background-daemon plan in `docs/background-service-plan.md`. v1.9.x is the perf-instrumentation arc — `viaduct::perf` log target with structured timing on every sidebar click, cancel-stale-fetch generation counter, bulk folder-aggregation DB op, and the v1.9.1 caps on `strip_html_for_preview` (400 chars) and `scan_html_for_video` (8 KB) that cured the "Digital Antiquarian feels slow" report. **v1.10.0 landed the Phase 17 background daemon**: window-hide-on-close + `run-in-background` GSetting + portal request via `ashpd` + D-Bus re-summon + idle-memory drop on hide. Phase 17's only remaining bullet is the Flathub tag (blocked on user — onboarding credentials, not code). **v2.0.0 ships the Phase 18 architectural refinement**: the three-pane decomposition (`ArticlePaneView` / `TimelineView` / `SidebarView` — pre1/pre2/pre3) shrank `window.rs` from 2900 to 2043 lines (−30 %); pre4 promoted `article_renderer.rs` into a dedicated `ViaductArticleRenderer` GObject with a per-renderer `WebContext` so the `viaduct-img://` + `viaduct-font://` scheme handlers no longer leak onto `WebContext::default()` (and into the video-dialog WebView); pre5 moved unread-count aggregation into `TreeNode` (folder/group totals auto-derive from leaves via `notify::unread-count` cascade); pre6 polished the WebKit ↔ GTK CSS bridge with `currentColor`-driven scrollbars, a 150 ms article-to-article fade-in, and `--accent-color` propagation that picks up the libadwaita system accent for the Adwaita theme. See `roadmap.md` for the live phase plan, `patchnotes.md` for the shipped log, and `docs/debugging.md` for the perf-log workflow.

## 3. Codebase Map

### Top-level

* `Cargo.toml` — **virtual workspace manifest** (since v1.5.0). Members: `viaduct-core`, `viaduct`. `default-members = ["viaduct"]` so `cargo run` at the repo root launches the GTK app. All shared dependency versions live in `[workspace.dependencies]` so the two crates can never drift out of sync. Key deps: `gtk4` v4.16, `libadwaita` v1.7, `webkit6` v0.4, `tokio` (full features), `rusqlite` (bundled + FTS5), `reqwest` (rustls-tls + gzip + brotli), `quick-xml`, `serde_json`, `ammonia`, `crossbeam-channel`, `thiserror`, `chrono`, `md-5`, `tracing`. `anyhow` is reserved for binary glue + the existing `network::credentials` / `error` adapters.
* `data/` — non-source assets shared across both crates. Holds `org.virinvictus.Viaduct.gschema.xml` (GSettings schema), `org.virinvictus.Viaduct.desktop` (freedesktop entry), `org.virinvictus.Viaduct.appdata.xml` (AppStream metainfo, installed as `<id>.metainfo.xml`), `themes/` (9 article themes), `fonts/` (Inter + SourceSerif4 + JetBrainsMono + atkinson/), and the build-produced `gschemas.compiled` (gitignored). Stays at the repo root; `viaduct/build.rs` walks `CARGO_MANIFEST_DIR.parent()` to find it for `glib-compile-schemas`, and the `include_str!` / `include_bytes!` calls in `viaduct/src/ui/article_renderer.rs` and `viaduct/src/fonts.rs` use three-up `../../../data/...` paths.
* `meson.build` (v1.5.1) — root-level meson wrapper. Drives `cargo build --release --package viaduct` via `custom_target`, installs the binary into `$prefix/bin/`, gschema into `$prefix/share/glib-2.0/schemas/` (with `glib-compile-schemas` run at install time), themes into `$prefix/share/viaduct/themes/`, desktop entry into `$prefix/share/applications/`, AppStream metainfo into `$prefix/share/metainfo/`. Flatpak manifest (`org.virinvictus.Viaduct.json`) uses `buildsystem: meson` so the build is shaped the way Flathub expects.
* `README.md`, `spec.md`, `roadmap.md`, `patchnotes.md` — authoritative design docs.
* `.github/workflows/ci.yml` — Ubuntu runner: `cargo fmt --all --check`, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo test --workspace`. Apt deps include `libwebkitgtk-6.0-dev`. All three checks must stay green.

### Crate split (v1.5.0)

The workspace has two members:

* **`viaduct-core/`** — **headless library**. No GTK / libadwaita / WebKit dependencies. Contains `database/`, `network/`, `parser/`, `models.rs`, `error.rs`, `paths.rs` plus the global Tokio runtime helpers (`init_runtime`, `spawn_on_runtime`, `block_on_runtime`) and debug-mode toggles. Its `Cargo.toml` deliberately omits `gtk`, `adw`, `webkit6`, and `tracing-subscriber` so reaching for them in this tree is a compile error. Contains the `tests/integration_refresh.rs` integration test.
* **`viaduct/`** — **binary crate**. Depends on `viaduct-core`. Contains `main.rs` (GTK entrypoint; **v2.6.12** declares `#[global_allocator] static GLOBAL: mimalloc::MiMalloc` at the top so the binary uses mimalloc instead of glibc malloc — flattens RSS drift across many refresh cycles by returning memory to the OS aggressively. `viaduct-core` stays allocator-agnostic. **v2.6.14** added `tune_mimalloc()` as the first call in `fn main` — sets `MIMALLOC_PURGE_DELAY=100` before any allocation so idle pages purge 10× faster than default. **v2.6.14** also added `crate::mimalloc_collect()` in `lib.rs` — direct FFI to `mi_collect(force=true)` so the refresh path can force a synchronous decommit at cycle end.), `ui/*` (every `!Send` GTK widget), `preferences.rs` (gio::Settings wrapper + adw::StyleManager hooks), `fonts.rs` (bundled-font installer that runs `fc-cache`), and `bin/mem_check.rs` (profiling harness). Its `lib.rs` re-exports `viaduct_core` symbols at the crate root (`pub use viaduct_core::{database, network, parser, models, error, paths, init_runtime, spawn_on_runtime, ...}`) so every existing `crate::xxx` import inside `ui/*` keeps working unchanged — no call-site churn.

### `viaduct-core/src/`

* `lib.rs` — headless library root. `pub mod database, error, models, network, parser, paths` + the global tokio runtime helpers (`init_runtime`, `spawn_on_runtime`, `block_on_runtime`) and debug-mode toggles (`is_debug_mode`, `set_debug_mode`, `spawn_debug_memory_ticker`). v2.6.3 promoted `read_memory_mb() -> (u64, u64)` from a private file-scope helper to `pub` so the binary crate can read VmRSS/VmHWM from the new `diag:` checkpoints (hide / re-show / refresh-cycle / hidden tick / tray Show+Quit) without duplicating the `/proc/self/status` parser.
* `paths.rs` — XDG resolution (post-v1.5.0: pure path resolution, no font side effects). Data → `$XDG_DATA_HOME/viaduct/` (fallback `~/.local/share/viaduct/`). Cache → `$XDG_CACHE_HOME/viaduct/` (fallback `~/.cache/viaduct/`). Resolves `local.opml`, `articles.sqlite`, `feed-settings.sqlite`, `sync.sqlite`, `favicons/`, `images/`, `video-thumbs/`, `fonts/viaduct/`. `ensure_dirs()` creates each of those directories and stops there. Bundled-font installation moved to `viaduct/src/fonts.rs`.
* `models.rs` — domain types: `Feed`, `Folder`, `Article`, `ArticleStatus`, `Author`, `FeedSettings`, `ParsedItem`, `ParsedFeed`, `ArticleChanges { new, updated, deleted }`.
* `error.rs` — `ViaductError` (top-level) → `DatabaseError`, `NetworkError`, `ParseError`. All via `thiserror`. Each variant preserves its source error (`rusqlite`, `reqwest`, `quick_xml`, `serde_json`, `url::ParseError`).
* `smart_feeds.rs` (v2.7.0) — **user-defined saved searches** (Custom Smart Feeds). `SmartFeed { id, name, rules: SmartFeedRules, created_at }`; `SmartFeedRules { conditions }` are AND-combined. `Condition` enum (8 variants): `TitleContains` / `BodyContains` / `AuthorContains` / `FeedIs` / `Read` / `Starred` / `NewerThanDays` / `OlderThanDays`. JSON-persisted via `SmartFeedsFile` (version-tagged; `load(path)` returns empty when missing, `save(path)` is atomic temp+rename). SQL compiler: `condition_sql(&Condition) -> (String, Vec<Value>)` per-condition fragment; `build_where(&rules)` joins fragments with `AND` and renumbers `?N` placeholders so they can't collide; `like_pattern` escapes LIKE wildcards. Consumed by `Account::{list,add,delete}_smart_feed` and the sidebar's "My Smart Feeds" group; UI in `ui::smart_feed_dialog`.
* `tests/integration_refresh.rs` — end-to-end refresh pipeline integration test. Uses `viaduct_core::*` imports (the test crate sees the headless side, not the GTK side).

### `viaduct/src/`

* `lib.rs` — binary crate's library root. Re-exports every `viaduct_core` module + runtime helper at the crate root via `pub use viaduct_core::{database, error, models, network, parser, paths, init_runtime, spawn_on_runtime, block_on_runtime, is_debug_mode, set_debug_mode, spawn_debug_memory_ticker, read_memory_mb}`. Then declares `pub mod fonts; pub mod preferences; pub mod tray; pub mod ui;`. Net effect: every `crate::xxx` reference inside `ui/*` resolves correctly whether `xxx` lives in core or in this crate; `crate::read_memory_mb()` is the shorthand the v2.6.3 `diag:` log lines use.
* `main.rs` — GTK entrypoint. Initializes tracing (`EnvFilter`, `RUST_LOG` controls verbosity), calls `ensure_schema_dir` to point `GSETTINGS_SCHEMA_DIR` at the workspace root's `data/` for dev builds (walks `CARGO_MANIFEST_DIR.parent()` because the binary crate lives one level down), boots the global tokio runtime via `viaduct::init_runtime`, calls `paths::ensure_dirs()` then `fonts::install_bundled()`, constructs `Account`, launches the `adw::Application`. `build_ui` first scans `app.windows()` for an existing `ViaductWindow` (Phase 17 D-Bus re-summon path — if the user has run-in-background on, hidden the window, and clicked the dock icon, GApplication routes the activation here and we want to `present` the same instance instead of building a second one) and calls `reload_current_timeline` so the user lands back on their selected feed with any articles fetched while hidden; only when no window exists does it apply the color-scheme preference and construct a fresh `ViaductWindow`. Uses the library via `use viaduct::…` (where `viaduct::database` etc. resolve via the lib.rs re-exports).
* `fonts.rs` — bundled-font installer (extracted from `paths.rs` in v1.5.0). `install_bundled()` writes the five bundled TTFs (`Inter-{Regular,Bold}`, `SourceSerif4-{Regular,Bold}`, `JetBrainsMono-Regular`) to `~/.local/share/fonts/viaduct/` if missing, then runs `fc-cache -f` so the WebKit article pane can resolve them. * `theme.rs` (v2.9.x, Phase 20b) — **dark/light resolution over `org.freedesktop.portal.Settings`**, the `adw::StyleManager` replacement (spec.md §12.3). `init(Option<gio::Settings>)` does one synchronous `ReadOne` before the first frame (with a `Read` fallback for older portals, whose reply wraps the value in a *second* variant layer), subscribes to `SettingChanged`, and holds the `gio::DBusConnection` in a thread-local so the subscription outlives the call; it is guarded against a second subscription because `build_ui` runs per activation. `is_dark()` is the drop-in for `adw::StyleManager::default().is_dark()`. **`re_resolve` is the single funnel and the reason this is not a raw portal read**: `is_dark()` returns the system value *folded with the user's `color-scheme` preference*, because `notify::dark` fired for a `set_color_scheme` call as well as a system flip, so a listener watching only the portal would miss the user picking Force dark. Flips are deduplicated. `connect_dark_changed(owner, f)` is the `connect_dark_notify` replacement: entries hold a **weak** owner and prune on each flip, closing the leak the Colophon pilot's migration exposed (closures on the `StyleManager` singleton were never disconnected); `broadcast` moves the registry out before invoking so a listener may register another without a re-entrant `borrow_mut` panic. **Two deliberate divergences from the pilot:** the no-portal default is **light** (the pilot degrades to dark because its palette is dark; adw resolves light, which is what viaduct does today), and the preference is folded in per above. Consumers: `article_pane_view::render_article_body` and `preferences::refresh_accent`. `article_renderer::system_accent_hex` deliberately still uses `adw` since it is inseparable from `accent_color()` and dies with libadwaita. **The write side (`update_style_manager`'s `set_color_scheme`) still drives widget styling until the 20c toolkit cut**, so both run side by side; they read the same portal and fold the same preference, so they agree.
* `preferences.rs` — `gio::Settings` wrapper over `data/org.virinvictus.Viaduct.gschema.xml`. `settings()` returns `Option<gio::Settings>` (None when the schema isn't installed — dev environment without `glib-compile-schemas`). **Process-singleton via `thread_local OnceCell` (v1.2.0)**: every call returns the same GObject so `connect_changed` handlers registered through any callsite stay alive past the caller's stack frame. Without this v1.2.0-pre1 shipped a non-functional theme picker — the dropdown wrote correct values to dconf but every listening Settings instance had been freed. `apply_color_scheme(&settings)` drives `adw::StyleManager::default().set_color_scheme` and connects `notify::color-scheme` for live updates; `notifications_enabled(&settings)` reads `notifications-on-refresh` on demand; `retention_days(&settings)` reads `retention-days` clamped to `[1, 365]` and is consumed by `act_refresh`/`refresh_specific_feeds` on the GTK thread before handoff to tokio (`gio::Settings` is `!Send`). **`apply_article_theme_accent(&settings)` (v1.2.0)** plumbs the user's `article-theme` GSetting through `resolve_article_theme(&settings, is_dark)` and pushes the chosen theme's `accent_hex` into `article_renderer::apply_app_accent`. Connects both the GSetting change notify AND `adw::StyleManager::dark_notify` so flipping system dark mode auto-swaps the accent in `auto` mode. Schema id: `org.virinvictus.Viaduct` (matches `Application::application_id`). Schema path: `/org/virinvictus/Viaduct/`.
* `tray.rs` (v2.5.0 + v2.6.6) — `ksni`-backed StatusNotifierItem, lit up whenever the `run-in-background` GSetting is on. Owns `wire(app)` (called once from `main.rs build_ui`), `start_service` / `stop_service` (gated on the GSetting via `connect_changed`), and a `receive_loop` that turns `TrayAction` (Show / Quit) into `app.activate()` / `app.quit()` on the GTK thread. **v2.6.6 icon pipeline**: `Tray::icon_pixmap` returns ARGB32 pixmaps decoded once per process from `docs/icon-{256,512}.png` (embedded via `include_bytes!`, ~42 KB compressed). Decode runs on the GTK thread via `gtk::gdk_pixbuf::PixbufLoader` (already a transitive of gtk4 — no new dep) and is cached in a `OnceLock<Vec<ksni::Icon>>`; the resulting `Icon` (plain `Vec<u8>` + `i32`s) crosses freely to the ksni tokio worker. RGBA → ARGB32 in network byte order via `pixel.rotate_right(1)` per the SNI spec. `icon_name` stays in place so themable hosts (KDE on a system with the SVG installed) can still pick it up. **v2.6.7 GNOME path**: `Tray::icon_theme_path` returns `$XDG_CACHE_HOME/viaduct/tray-icons/`, populated at startup with the same PNG bytes laid out as `hicolor/{256x256,512x512}/apps/org.virinvictus.Viaduct.png`. Required because the GNOME AppIndicator Shell extension mostly ignores `IconPixmap` and resolves through GTK's icon theme via `IconName` + `IconThemePath`. Install is idempotent (size-compare guard) and cached behind a second `OnceLock<Option<String>>`. **v2.6.8** also writes `<base>/hicolor/index.theme` declaring the size buckets — without it, `St.IconTheme.lookup_icon_for_scale` can't enumerate the subdirs and the lookup falls back to the placeholder despite the PNGs being on disk. v2.6.3 added `diag: tray Show` / `diag: tray Quit` `tracing::info` lines on each menu activation.
* `bin/mem_check.rs` — Phase 7 + Phase 10 + Phase 17 memory checkpoint harness. Synthesizes a 500-feed × 10-article corpus in a tempdir XDG, runs `LocalAccount::update_feed` through the real single-writer worker, warms the favicon and image cache against an in-process `tokio::net::TcpListener` HTTP/1.1 fixture (500 favicons × 1 KB + 50 images × 50 KB, path-prefix routed `/fav-*` vs `/img-*`; note since v2.8.0 the cache is byte-bounded, not count-capped, so 500 × 1 KB favicons stay resident under the 16 MB favicon ceiling rather than exercising eviction), then runs `ui::reader_view::extract` 10× over a synthesized ~100 KB article HTML laden with chrome/sidebar/ads so the readability scoring path actually fires, then exercises the Phase 17 background-cycle path by calling `ImageCache::clear_memory_now()` and reading the RSS delta. Reads `VmHWM` / `VmRSS` from `/proc/self/status` at four checkpoints (post-DB, post-warmup, post-reader-view, post-background-cycle) and pass/fails against the 500 MB peak budget. `cargo run --release --bin mem_check`. Current release-build numbers: post-DB peak ~36 MB, post-warmup peak ~56 MB, post-reader-view peak ~61 MB, post-clear delta ~−4 MB on the synthetic corpus. The full GUI-side hide cycle (WebView idle, ListStore compact) needs interactive QA — those widgets can't be constructed from a headless bin. The harness installs the global tokio runtime via `viaduct::init_runtime` so `ImageCache`'s `spawn_on_runtime` calls resolve correctly.

### `viaduct-core/src/database/`

Three stores, strict separation, single writer. Port of NNW's three-way split.

* `articles.rs` — `ArticlesDatabase`. Tables: `articles` (with `attachments JSON` column added by an idempotent ALTER for pre-existing DBs), `statuses`, `authors`, `authorsLookup` (with `authorsLookup_article_id_idx` index, v1.5.2 — speeds up both the delete trigger and the orphan sweep), FTS5 virtual `search`, and `db_info` (v2.8.2 key/value table; port of NNW `4c85c907f` `RSDatabaseInfoTable` — holds `last_vacuum_date` for the VACUUM throttle, read/written by the `LastVacuumDate` op + `stamp_vacuum_date`). Triggers: FTS index maintenance on insert/update/delete, and `articles_ad_lookup` to cascade-clean `authorsLookup` when an article is removed (status rows are deliberately NOT cascaded — NNW keeps them for reappearing articles). WAL + `synchronous=NORMAL`, `temp_store=MEMORY`. **v2.6.11**: `mmap_size = 67108864` (64 MB; pre-v2.6.11 was 30 GB which let SQLite map the entire DB + WAL into RSS) and `journal_size_limit = 67108864` (64 MB; pre-v2.6.11 was uncapped, so a 130-feed force-refresh ballooned WAL to 149 MB and added ~60 MB to per-cycle RSS). Public helpers: `article_id_for(feed_id, unique_id)` — MD5 of `"{feed_id} {unique_id}"` per NNW `Article.calculatedArticleID`; `parsed_to_article` which truncates `DateTime` to second precision so DB round-trips don't flag every article as "updated"; `pub const DEFAULT_RETENTION_DAYS: i64 = 30` for callers without a GSettings handle. Main op is `ArticlesDbOp::UpdateFeed { feed_id, items, delete_older, retention_days, reply }` which produces a real `ArticleChanges` diff (new / updated / deleted + status rows for new articles; stale items >6 months default to `read=1`; orphans non-starred older than `retention_days` get deleted when `delete_older=true`). Phase 14 prune ops: `DeleteArticlesNotInFeeds(feed_ids)` (empty input is a no-op, matches NNW `deleteArticlesNotInSubscribedToFeedIDs`), `DeleteOldStatuses { retention_days }` (NNW `feedBased` branch — `WHERE date_arrived < ? AND starred = 0 AND article_id NOT IN (SELECT article_id FROM articles)`), `DeleteOrphanedAuthors` (v1.5.2; NNW 200e5b19f port — sweeps `authorsLookup` rows whose article no longer exists then drops `authors` rows no longer referenced by any lookup; the trigger handles live deletes, this op is the startup safety net), `Vacuum`, `LastVacuumDate` (reads `db_info.last_vacuum_date` for the v2.8.2 13-day VACUUM throttle). Other ops: `Search`, `SearchWithSnippets` (FTS5 `snippet()` + optional feed-scope filter), `FetchStatusesByIds` (bulk status lookup for keyboard navigation, chunked at 500 IDs to stay under SQLite's parameter limit), `UnreadCountsByFeed` (per-feed unread totals via `LEFT JOIN statuses` so missing-status rows count as unread per NNW), `SmartFeedCounts` (returns a `SmartFeedCounts { today_unread, all_unread, starred_unread }` struct used by the sidebar's three pinned rows).
* `settings.rs` — `FeedSettingsDatabase`. Per-feed cache: `feed_id` (PK, string — note NNW uses `feedURL` as PK, we diverge), `feed_url`, `home_page_url`, `icon_url`, `favicon_url`, `edited_name`, `content_hash`, conditional-GET (`etag`, `last_modified`, `date_created` = when those were received), cache-control (`max_age`), `authors_json`, `folder_relationship_json`, `last_check_date`, `reader_view_always_enabled`, `new_article_notifications_enabled`, and `last_response_code` (v2.8.2, NNW `4c85c907f` `lastResponseCode` port — the HTTP status of the last download attempt; added by an idempotent ALTER for pre-existing DBs and persisted by the refresher). `delete_settings_for_feeds_not_in` early-returns on empty input (regression-tested) — do NOT "simplify" that branch back to a bare DELETE or you'll wipe every row.
* `opml.rs` — OPML on disk is the source of truth for the feed/folder hierarchy. Coalesced save (~500ms debounce), atomic temp-file + rename. `OpmlWriter::spawn` owns its own tokio task; `save(OpmlFile)` queues and awaits the next flush. Phase 12 user-import pipeline lives here too: `normalize_opml` (port of NNW `OPMLNormalizer` — drops nameless wrappers, flattens nested folders one-level-deep, dedups feeds by `xmlUrl`), `merge_opml(existing, incoming) -> (OpmlFile, Vec<Feed>)` (union by `xmlUrl`, never overwrites `edited_name`, returns just the newly-added feeds for refresh), and `serialize_account_opml(title, &OpmlFile)` (hand-rolled NNW-shape writer matching `OPMLExporter.OPMLString` byte-for-byte: tab indent, attribute order, `description=""`, `version="RSS"`, `<!-- OPML generated by viaduct -->` comment). The serde-driven `serialize_opml` continues to back the on-disk debounced save.
* `accounts.rs` — `LocalAccount` orchestrator owning the OPML file + both DBs. Public async API: `load_opml`, `save_opml`, `batch_insert_articles`, `upsert_statuses`, `fetch_articles_by_feed`, `fetch_unread_articles`, `fetch_starred_articles`, `fetch_today_articles`, `search_articles`, `search_articles_with_snippets`, `fetch_statuses_by_ids`, `unread_counts_by_feed` (sidebar badges), `smart_feed_counts` (Today/All Unread/Starred totals), `fetch_feed_settings`, `upsert_feed_settings`, `update_feed(feed_id, items, delete_older, retention_days)` (the diff path; `retention_days` flows through to the per-update prune), `cleanup_orphaned_settings`, `cleanup_at_startup(retention_days)` (Phase 14 chain + v1.5.2 + v2.6.5 — settings prune → article prune → status sweep → orphan-author sweep → **disk-cache sweep** (v2.6.5: targeted favicons + age-based images/thumbs via `cache_sweep`) → **local-only `syncStatus` wipe** (v2.6.5: when `is_local_account()`) → vacuum on both DBs; v2.6.11 prepends `PRAGMA wal_checkpoint(TRUNCATE)` to vacuum so the existing oversized WAL gets reclaimed once on upgrade; **v2.8.2** gates the full VACUUM on prune activity AND `>= VACUUM_INTERVAL_DAYS` (13) since the last one via `last_vacuum_date()` + the articles `db_info.last_vacuum_date` stamp — a non-regressing hybrid of our v2.8.0 prune gate and NNW `4c85c907f`'s `vacuumIfNeeded` throttle), `vacuum_databases`, `last_vacuum_date`, `import_opml(path) -> Vec<Feed>` (parses → normalizes → merges → saves, returns newly-added feeds for refresh), `export_opml(path, title)` (writes the NNW-shaped string). `LocalAccount::new` runs `cleanup_at_startup` instead of just the settings sweep. Private helpers `opml_feed_urls` and `opml_feed_ids` walk standalone+folder feeds for the cleanup methods.
* `worker.rs` — **single writer + read-only pool**. The writer: `std::thread::spawn` pulls `DbOp`s off a tokio `mpsc::Receiver` via `blocking_recv` and dispatches to `articles::handle_op` or `settings::handle_op`. The GTK thread holds only a `Sender`; it never blocks on SQLite. **Panic supervisor**: both the writer and sync workers run their `recv` loop inside `std::panic::catch_unwind` and restart it (1 s backoff) on a panic, so one bad op can't take the DB layer down for the session. **Read pool (v2.8.0)**: `spawn_read_workers` spawns `READ_POOL_SIZE = 3` threads, each owning its own read-only (`OpenFlags::SQLITE_OPEN_READ_ONLY`, `busy_timeout = 5 s`, no mmap) articles connection, pulling `ArticlesDbOp`s off a shared `crossbeam_channel::Receiver`. WAL lets these readers run concurrently with the writer, so timeline fetches / searches / counts no longer queue behind a long write. `read_channel()` makes the pair; `Account` holds the `Sender` as `Option` (`None` → reads fall back to the writer, used by tests / `mem_check`). The reader threads share the same panic-supervisor pattern and **must** be spawned after `Account::new` returns so the writer has already created + WAL-configured the file. Account routes only articles **reads** to the pool; every write and all settings ops stay on the single writer.
* `delegate.rs` — **account-backend strategy** (the `AccountDelegate` trait: `refresh_all` / `sync_article_status` / `import_opml` / `is_local`). Two impls: `LocalAccountDelegate` (no-op sync, `is_local() = true`) and `InoreaderAccountDelegate`, which drives the four-phase Inoreader sync — reconcile subscriptions + tags via OPML, push local read/starred deltas, pull remote status deltas back, then fetch missing article bodies in 100-item chunks through `ReaderAPICaller::retrieve_entries` — pulling the token from the Secret Service via `credentials::fetch_credentials("inoreader")`. Instantiated per account in `Account::new` and invoked on the tokio runtime. **v2.8.3 (NNW `08d10f501`)**: the status-send step is log-and-continue, never `?`. A failed send used to abort `refresh_all`, so the status pull and article fetch never ran and the account silently stopped updating. Failed rows stay queued and step 3's `reset_all_sync_statuses_selected_for_processing` re-arms them for the next cycle. Only articles whose *every* batch landed are deleted, because `syncStatus` deletes by `articleID` alone and clearing a half-sent article would drop the row for its other key. NNW counterpart: the `AccountDelegate` protocol + `LocalAccountDelegate` / Reader-API delegate.
* `sync.rs` — **syncStatus queue** (Inoreader only; port of NNW `SyncDatabase`). `SyncStatus { article_id, key ("read"/"starred"), flag, selected }` tracks pending local→remote status deltas. `SyncDbOp`: `InsertStatuses`, `SelectForProcessing` (mark `selected=1`), `DeleteSelectedForProcessing`, `ResetAllSelectedForProcessing` (rollback an interrupted cycle), `WipeAll` (v2.6.5 — clears ghost rows when an account switches back to local). `setup_schema` + transactional `handle_op`, run on the dedicated sync worker thread in `worker.rs` under the same panic supervisor as the article/settings writers.

### `viaduct-core/src/network/`

* `fetcher.rs` — `Fetcher` (reqwest, rustls-tls, HTTP/2 auto-negotiated, UA `Viaduct/1.0 (Linux; GTK4)`, `pub fn client(&self) -> Client` accessor since v2.6.4 so favicon / feed_discovery share the connection pool) + `DownloadSession` analog that coalesces duplicate in-flight URLs via `broadcast::channel` and tracks per-host cooldowns after 429. **v2.8.3 (NNW `c9bd65b1f`)**: the cooldown is now recorded on *every* 429 via the free fn `retry_after_secs`, falling back to `DEFAULT_RETRY_AFTER_SECS` (10 min) when `Retry-After` is missing, unparseable (an HTTP-date is legal and we don't parse it), or non-positive. The old `&&`-chain short-circuited on an absent header and recorded no cooldown at all, so hosts that 429 without the header (Reddit) got hit again next cycle. **v2.6.9**: `refresh_feeds_inner` gates per-feed pipelines on a `tokio::sync::Semaphore` capped at `REFRESH_PARALLELISM = 8` so a 130-feed cycle no longer holds 130 HTTP bodies + 130 parsed feeds + 130 favicon-discovery HTML pages in memory simultaneously (the v2.6.3 `diag: refresh cycle` log line caught `peak_delta_mb=124` pre-fix in a real user run; bounded version should be ~16× lower per cycle). **v2.6.10**: `AccountRefresher::with_completion_counter(Arc<AtomicUsize>)` accepts an opt-in counter; the per-feed task tail bumps it on every completion (success / 304 / fetch error / parse error all count) and the skip branch bumps it too, so the GTK-side progress bar in `window.rs` reaches 100 % cleanly regardless of how many feeds got throttled. `LocalAccountRefresher::new(account: Arc<LocalAccount>, changes_sender, retention_days: i64)` — `retention_days` is forwarded to `account.update_feed` so the per-feed orphan sweep honors the user's `retention-days` GSetting. The refresher pipeline actually does parse + diff + persist: conditional-GET headers from `FeedSettingsDatabase`, short-circuits on 304 (updates `last_check_date` only), content-hash short-circuits on byte-identical bodies, calls `parser::parse` → `account.update_feed` → emits real `ArticleChanges`, persists ETag/Last-Modified/`content_hash`/`last_check_date`/`max_age`/`last_response_code` (v2.8.2 — `result.status` recorded for every HTTP response: 304, non-200, and success alike) + (v2.6.4) `home_page_url` from `parsed.home_page_url` back to settings, and (v2.6.4) when `favicon_url` is currently None calls `favicon_discovery::discover_favicon(&fetcher.client, home)` and persists the result so the probe runs at most once per feed across the install lifetime. Implements NNW's **8-day conditional-GET expiry** (catches servers that always 304; openrss.org + rachelbythebay.com are exempt), **5-hour cap** on `Cache-Control: max-age` (openrss.org exempt), and **25-hour special-case cutoff** for high-frequency feeds. Skip rules: twitter.com/x.com, cache-control freshness window, then (v2.8.1) a `settings.max_age.is_some()` exemption so Cache-Control feeds bypass the minimum-time clamp, else the 9-min minimum between checks (`MINIMUM_TIME_BETWEEN_CHECKS_MINUTES`, lowered from 29 in v2.8.1). **Domain matching** via `url_host_matches_domain(url, &[&str])` — port of NNW `SpecialCase.urlStringMatchesDomain`. Parses URL → lowercases host → strips optional `www.` → matches a domain on an exact equality OR a `.`-prefixed suffix (**v2.8.2, NNW `4c85c907f`**: `gruber.micro.blog` now matches `micro.blog`; empty domains are skipped). Plain substring matching (the original behavior) was a real bug: `evilrachelbythebay.com` and `rachelbythebay.com.evil.com` would have false-matched, but the suffix form rejects both. Three module-level constants drive the policy: `SPECIAL_CASE_DOMAINS` (rachelbythebay.com, openrss.org — get the 25-hour cutoff and conditional-GET exemption), `NO_MINIMUM_TIME_DOMAINS` (19 personal-site hosts that skip the 9-minute minimum entirely; synced from NNW `LocalAccountRefresher.domainsWithNoMinimumTime` as of `4d594181f`), and the `is_no_minimum_time_domain` short-circuit runs **first** in the timing skip check, mirroring NNW's order.
* `cache.rs` — `ImageCache` with two-tier storage: in-memory `ByteLru` per kind (favicons + images + video thumbs, each a separate **byte-bounded** LRU — v2.8.0, ceilings 16/64/16 MB; replaced the old 250-entry count cap so a run of large images can't pile up past the budget), disk at `$XDG_CACHE_HOME/viaduct/{favicons,images,video-thumbs}/<md5-of-url>`. **Per-download size caps** (v2.8.0): `download` streams the body with a running guard + Content-Length pre-check, dropping anything over 1 MB (favicon) / 20 MB (image) / 4 MB (thumb). Deliberately stores `Vec<u8>` (not `GdkTexture`) so the LRU stays `Send`; decode to texture happens on the GTK main thread at the call site. Public API: `favicon(url)`, `image(url)`, `video_thumbnail(url)` (v1.3.0), `client()` (returns the underlying reqwest `Client` so callers can issue oEmbed lookups before fetching the actual thumb), `color_for(s)` (port of NNW `ColorHash`, returns `#rrggbb` from MD5 of input), `clear_memory()` / `clear_memory_now()` (Phase 17: drop all three in-memory LRUs while leaving the disk cache intact, used by `ViaductWindow::hide_for_background` when the user closes the window in run-in-background mode; the `_now` variant is awaitable for the `mem_check` harness). `Kind` enum has three variants: `Favicon`, `Image`, `VideoThumb`.
* `activity.rs` (v2.6.24) — in-memory activity ring buffer for the refresh pipeline. `ActivityLog` holds `Arc<RwLock<VecDeque<ActivityEvent>>>` capped at `RING_CAPACITY = 500`; pushes drop the oldest entry on overflow. `ActivityEvent { at, feed_id, feed_url, feed_name, kind }` carries terminal-state context. `ActivityKind` enum: `Success { new, updated, deleted }`, `NotModified`, `HttpError { status }`, `NetworkError { detail }`, `ParseError { detail }`, `DbError { detail }`, `Skipped(SkipReason)`. `SkipReason` is `DisallowedHost | CacheControl | Throttled`. `snapshot()` returns events newest-first (clones the ring under read lock). 3 unit tests covering ring eviction, snapshot ordering, and clear. Wired into `AccountRefresher` via `with_activity_log(log)` opt-in builder; `refresh_one_feed` pushes one event per terminal site; the pre-pipeline skip branch in `refresh_feeds_inner` pushes via the `skip_reason -> Option<SkipReason>` helper that replaced the prior bool-returning `feed_should_be_skipped`.
* `cache_sweep.rs` (v2.6.5) — disk-cache garbage collection. Three free functions: `sweep_targeted(dir, live_filenames)` for favicons (caller supplies the live md5 set from `feed_settings`); `sweep_by_age(dir, max_age_days)` for images / video-thumbs (mtime-gated since `noatime`/`relatime` mounts make atime unreliable); `wipe_dir(dir)` for the `--debug` "Wipe Disk Caches" action. Plus `live_filenames_for(urls) -> HashSet<String>` that hashes through the same `cache::cache_filename` (now `pub(crate)`) the writer uses, so the live-set computation can never drift from the on-disk naming. Called from `Account::cleanup_at_startup` as a fifth step after the existing article/status/author sweeps. 8 unit tests covering targeted/age/wipe + missing-dir handling + live-set round-trip.
* `favicon_discovery.rs` (v2.6.4) — port of NNW's `SingleFaviconDownloader` flow for sidebar favicons. `discover_favicon(client, home_page_url) -> Option<String>` two-pass resolver: GET the home page (256 KB cap), scan the head with `parser::extract_metadata`, pick the best `<link>` candidate (plain `rel="icon"` / `rel="shortcut icon"` first, `rel="apple-touch-icon"` as fallback), HEAD-verify with GET fallback (some Cloudflare-fronted hosts 405 HEAD), then fall back to `<origin>/favicon.ico`. Refresher persists the resolved URL into `FeedSettings.favicon_url`; sidebar's `spawn_favicon_fetch` (`favicon_url.or(icon_url)`) renders it. 9 unit tests covering each rel variant + relative-href resolution + ignore-non-icon-links.
* `feed_discovery.rs` (v1.7.0) — port of NNW's `FeedFinder` for the Add Feed dialog. `discover_feed(client, url) -> Result<DiscoveredFeed>` two-pass resolver: parse the URL's content directly as a feed first (RSS / RDF / Atom / JSON Feed dispatch through `parser::parse`); on parse failure, scan HTML head via `parser::extract_metadata` for `<link rel="alternate" type="application/{rss,atom}+xml | feed+json">` and recurse into the first match. Recursion depth capped at 2 to bound the loop. URL canonicalization adds `https://` for bare hostnames. 8 unit tests covering each branch. Returns `NetworkError::NoFeedFound` when both passes fail.

### `viaduct-core/src/parser/`

* `xml.rs` — RSS 2.0, Atom, OPML via `quick-xml`. **Entity decoding is delegated to `quick_xml`'s `unescape()` rather than a ported table (NNW's `RSParser/XML/XMLEntities.swift`), which makes the `escape-html` feature in the workspace `Cargo.toml` load-bearing, not cosmetic: without it `unescape()` resolves only the five predefined XML entities and returns `Err` on every named HTML one, and because both parsers spell the call `let Ok(text) = e.unescape() else { continue }`, the text was *dropped* — an item titled `Tom &mdash; Jerry` parsed to `title: None`. Fixed in v2.8.3, which is also how NNW `3a948b755` (`&zwj;` / `&zwnj;`) landed. Regression-tested by `html_named_entities_decode_rather_than_dropping_the_text` + `zero_width_entities_decode`; do not drop the feature.** RSS handles: RDF (`rdf:RDF`), `<content:encoded>`, `<dc:creator>`/`<dc:date>` via unprefixed local-name match, `<guid isPermaLink="false">` attribute, relative-URL resolution against home-page URL, `</rss>`/`</RDF>` stop-sentinel, MD5 synthetic IDs, RSS `<enclosure>` and MRSS `<media:content>`/`<media:thumbnail>` → `Article.attachments`, channel `<image><url>` → `ParsedFeed.icon_url`, `<language>` → `ParsedFeed.language`. Atom handles: `in_author`/`in_source` state tracking (a `<source>` block inside an entry does NOT overwrite the entry's fields), `<author><name>`/`<email>`/`<uri>` capture into `MutableAuthor`, feed-level root author propagated to authorless entries at end of parse, `<link href>` resolution against home-page URL with `AtomLinkRel` (alternate/related/enclosure/other), `</feed>` stop-sentinel, `<icon>`/`<logo>` (icon wins) → `ParsedFeed.icon_url`, `<feed xml:lang>` → `ParsedFeed.language`, `<link rel="enclosure">` → `Article.attachments`, **`type="xhtml"` raw inner HTML capture for `<content>`/`<summary>`** via `capture_atom_xhtml_inner` (re-serializes inner XML through `quick_xml::Writer` with `trim_text(false)` scoped around the capture so inline whitespace survives — port of NNW `XMLSAXParser.captureRawInnerContent`). **CDATA bodies**: both parsers handle `Event::CData` alongside `Event::Text` via `rss_handle_text_or_cdata` / `atom_handle_text_or_cdata` helpers — Sacha Chua's blog and most WordPress / Hugo / Jekyll feeds wrap their `<description>` and `<content:encoded>` payloads in CDATA, and the parser dropped them silently before v1.0.8. **Namespaced-section skip (v2.8.0, NNW #4422):** both parsers carry a `namespace_element_depth` counter so that inside an item/entry a prefixed child element and its whole subtree are swallowed, preventing a nested unprefixed `<title>` (e.g. Shopify's `<s:variant>`) from being read as the item's own. RSS keeps `dc:`/`content:`/`media:` live via `rss_prefix_is_consumed` (wider than NNW's dc/content/source because we parse MRSS); Atom skips every prefixed child (its own elements are all unprefixed).
* `json.rs` — JSON Feed + RSS-in-JSON via `serde_json`. MD5 synthetic IDs (not `DefaultHasher` — must stay stable across builds).
* `html.rs` — `HTMLMetadataExtractor` (finds `<link rel="alternate" type="application/rss+xml|atom+xml">` when a user pastes a bare website URL); returns an `HtmlMetadata { url_string, tags }` bag. HTML sanitization for the reading pane lives in `src/ui/article_renderer.rs` via `ammonia::Builder` with the `viaduct-img://` URL scheme allowlisted.
* `date.rs` — `DateParser` port (NNW `RSDateParser`). Permissive: W3C / ISO 8601, RFC 822 / `pubDate`, and the long tail of malformed real-world dates. Byte-level inspection for zero-alloc parsing; `chrono` for the final conversion.

## 4. NetNewsWire Porting Philosophy

viaduct is a **translation**, not a reimagining. This is the most important section of this file.

### The Rule

Port. Don't invent.

When implementing anything — a parser, a queue, a refresher, a cache eviction policy, an error-handling branch — find the corresponding code in `.netnewswire/` and translate it to Rust. Match the structure. Match the names (adapted to Rust conventions). Match the edge cases, including the ones that look weird. NetNewsWire has been shipped and maintained since 2002 — the weird-looking branches are almost always there because someone's feed broke without them.

**Do not:**
* Invent your own architecture because the Swift approach "isn't idiomatic Rust."
* Substitute a crate with different semantics because it's more popular.
* Skip porting a branch because you can't immediately think of when it fires.
* "Improve" anything on the first pass. Make it work *the NNW way*, then we discuss.
* Build a feature from scratch when NNW already has it. If you can't find it, ask — it's probably there and you missed it.

### Workflow

1. Identify the NetNewsWire file(s) that implement the feature. Read them.
2. Translate to Rust, preserving structure and behavior.
3. Adapt only where Swift/Apple primitives have no direct analog (see §4, "Unless It Has To").
4. Verify behavior matches. Then we ship.

Ambiguous user request → go read `.netnewswire/`. That is the default plan.

### What NOT to Port

**Only `.netnewswire/Modules/` and `.netnewswire/Shared/` matter for logic porting.** The rest of the NNW tree is platform-specific UI we're replacing with GTK — ignore it entirely:

* `.netnewswire/iOS/`, `.netnewswire/Mac/`, `.netnewswire/Widget/` — AppKit / UIKit views, view controllers, storyboards. We are building a native GTK4 UI from scratch. Don't port these.
* `.netnewswire/Intents/`, `.netnewswire/AppleScript/` — Siri Shortcuts and AppleScript bindings. No Linux analog; skip.
* `.netnewswire/AppStore/`, `.netnewswire/Appcasts/` — distribution plumbing for Apple platforms. Irrelevant; we ship Flatpak.
* `.netnewswire/buildscripts/`, `.netnewswire/xcconfig/`, `.netnewswire/NetNewsWire.xcodeproj/` — Xcode build system. Irrelevant.

**viaduct supports local accounts and Inoreader.** We are not porting other remote-account or sync code. This is a hard scope boundary, not a "we'll get to it later."

Explicitly skip the following directories/modules when porting:

* `.netnewswire/Modules/CloudKitSync/` — iCloud sync. Skip entirely.
* `.netnewswire/Modules/NewsBlur/` — NewsBlur API client. Skip.
* `.netnewswire/Modules/SyncDatabase/` — exists to support remote sync. Ported for Inoreader as `database/sync.rs` (the syncStatus queue).
* `.netnewswire/Modules/Secrets/` — holds credentials for remote services. Ported for Inoreader as `network/credentials.rs` (Secret Service via `oo7`).
* Anything inside `Modules/Account/` related to `FeedbinAccountDelegate`, `FeedlyAccountDelegate`, `FreshRSSAccountDelegate`, `NewsBlurAccountDelegate`, `BazQuxAccountDelegate`, `TheOldReaderAccountDelegate`. Port only `LocalAccountDelegate`, `InoreaderAccountDelegate`, and the shared account scaffolding they need.

When porting a file that mixes local-account/Inoreader logic with other remote-sync logic, take the local and Inoreader branches and drop the rest. Do not leave stubs, `todo!()` placeholders, or "future sync" interfaces — cut the other code out cleanly. If a generic abstraction exists solely to accommodate unsupported remote accounts, collapse it to the local and Inoreader shapes.

If you're unsure whether something is local-only/Inoreader or other remote-sync, ask. Don't port "just in case."

### Where to map

| Concept | NNW | viaduct |
|---|---|---|
| Parsing | `Modules/RSParser` | `src/parser/` |
| Dates | `RSDateParser` | `src/parser/date.rs` (`chrono`) |
| Articles store | `ArticlesDatabase` | `src/database/articles.rs` (`rusqlite`) |
| Settings store | `FeedMetadataFile` + per-feed metadata | `src/database/settings.rs` |
| Feed hierarchy | `OPMLFile` | `src/database/opml.rs` |
| Single-writer serialization | `DatabaseQueue` | `src/database/worker.rs` (tokio mpsc) |
| Coalescing UI updates | `CoalescingQueue`, `BatchUpdate` | tokio channels + debouncing |
| Refresh orchestration | `LocalAccountRefresher` | `src/network/fetcher.rs` |
| Download coalescing | `DownloadSession` | `src/network/fetcher.rs` |

### The "Unless It Has To" Rule

Deviate **only** when Swift/Apple primitives have no direct analog. Approved deviations:

* **Concurrency:** GCD / `@MainActor` → `tokio` multi-threaded runtime + `mpsc` for work, `glib::MainContext::channel` for hopping back to the GTK thread.
* **UI:** AppKit / UIKit view recycling → GTK4 `gio::ListModel` + `GtkListView` + `GtkSignalListItemFactory`.
* **Reader View:** NNW calls a remote Mercury service. We must run extraction locally (Phase 10), RAM-gated.
* **FTS:** NNW uses FTS4. We use FTS5.

That's the full list. If you think you need another deviation, stop and ask.

## 5. GNOME 50+ Exclusivity

Target stack: **GNOME 50+**, **GTK4 ≥ 4.16**, **libadwaita ≥ 1.7**, Rust 2024 edition.

* No polyfills, no fallback paths, no conditional compilation for older GNOME/GTK versions.

## 6. Data, Storage, and Thread Model (invariants)

* **Storage root:** `$XDG_DATA_HOME/viaduct/` holds `local.opml`, `articles.sqlite`, `feed-settings.sqlite`. `$XDG_CACHE_HOME/viaduct/` holds `favicons/` and `images/`.
* **OPML is the feed-hierarchy source of truth.** Not SQL. Do not add a `feeds` table with folder FK columns.
* **WAL mode** on both SQLite DBs, always. FTS5 on `articles`.
* **Single writer task.** All SQLite writes funnel through `src/database/worker.rs`. The GTK thread only ever sends commands or reads from in-memory models.
* **UI delivery:** background tasks emit `ArticleChanges { new, updated, deleted }` batches; these cross into the GTK main loop via `glib::MainContext::channel`. Bulk operations coalesce through a `BatchUpdate` primitive to avoid notification storms.
* **Article DB ID = `md5("{feed_id} {unique_id}")`.** Ported from NNW `Article.calculatedArticleID(feedID:uniqueID:)`. The `unique_id` is whatever the feed's `<guid>` / Atom `<id>` / JSON Feed `id` gave us, and the parser falls back to MD5 of a deterministic concatenation when that's missing. Under no circumstances use `DefaultHasher` for synthetic IDs — it's not stable across builds and will orphan status rows on every restart.
* **`parsed_to_article` truncates dates to whole seconds** before inserting, matching the integer column type. Don't remove this — without it, every refresh will flag every article as "updated" because the in-memory `DateTime<Utc>` has nanosecond precision that disagrees with the round-tripped seconds value.
* **`delete_settings_for_feeds_not_in(Vec::new())` must be a no-op.** NNW `guard !feedURLs.isEmpty else { return }`. We used to `DELETE FROM feed_settings` on empty input, which nuked the entire settings DB whenever startup OPML was empty. Regression-tested; don't "simplify" the early return away.
* **Status rows outlive articles.** The `articles_ad_lookup` trigger cascades `authorsLookup` deletes but `statuses` deletes are intentionally left out — if a feed re-adds an article after a retention sweep, NNW expects the old read/starred state to come back with it.

### gtk-rs Gotchas (Swift→Rust Pitfalls)

Swift's memory model is forgiving in ways Rust's isn't. Specifically, porting Swift code that freely captures `self` in closures will fight you here.

* **`GObject` is `!Send`.** GTK widgets cannot cross threads. Do not hold a `GtkWidget` inside a struct that will be passed to `tokio::spawn`. Move *data* across the boundary, not widgets. The pattern: background task sends `ArticleChanges` → GTK-thread handler receives it via `glib::MainContext::channel` → handler updates widgets on the main thread.
* **Use `glib::clone!` with weak refs.** A signal handler that captures `self` strongly creates a reference cycle (widget → handler closure → widget) that GTK will not clean up. Always prefer `glib::clone!(#[weak] self_ => async move { ... })` or `#[weak_allow_none]`. Strong captures (`#[strong]`) are acceptable only when you explicitly want the closure to extend the widget's lifetime and you've thought about it.
* **Tokio runtime lives outside the GTK loop.** Spawn background work with `tokio::spawn` against the runtime built in `main.rs`; spawn UI-touching work with `glib::spawn_future_local`. Never `.await` on a tokio future from a GTK signal handler — route through a channel instead.
* **Tokio I/O Requires a Reactor:** If you call Tokio network/filesystem APIs (`reqwest`, `tokio::fs`) from inside a `glib::spawn_future_local` block, they will panic with "there is no reactor running". GTK's async executor is not backed by Tokio. Always use `crate::spawn_on_runtime` to execute I/O work, then hand the result back to the GTK thread.
* **`glib::MainContext::channel` is one-way.** It delivers from a worker to the GTK loop. For GTK → worker, use the `mpsc::Sender` held by `src/database/worker.rs` (or equivalent). Don't try to make one channel work both directions.
* **Swift's `weak self` ≠ Rust's `Weak<T>`.** When porting a Swift closure that uses `[weak self]`, translate to `glib::clone!(#[weak] ...)` for GObject-rooted captures, or `Weak<RefCell<T>>` / `Arc::downgrade` for plain Rust types. Don't paper over it with `Arc` and hope.

## 7. Agent Operational Constraints

Non-negotiable unless the user overrides in conversation.

1. **Memory budget is supreme.** Idle: **100–300 MB** after full sync + image cache warm. Peak: **< 500 MB** across every supported operation. Any feature proposal that plausibly busts this is rejected on sight (this includes embedded WebKit, in-process image decoding of uncapped size, eager full-text extraction, etc.). Every major phase ends with a `heaptrack`/`massif` checkpoint.
2. **Never block the GTK thread.** All I/O (network, disk, SQLite writes, XML parsing, HTML sanitization) runs in tokio tasks. The main thread renders and reads — nothing else.
3. **Local + Inoreader accounts only.** Through v1.0 this rule was local-OPML-only with no scaffolding of any remote interface. Post-1.0, Inoreader is the one supported remote (per §4), and its delegate is wired into `Account::new`. Everything else stays rejected outright: Feedbin, Miniflux, FreshRSS, CloudKit, NewsBlur, Feedly, BazQux, TheOldReader. Do not scaffold interfaces for any of those "in case."
4. **Neutered WebKit Instance** (shipped v1.1.0). Exactly ONE `WebKitWebView` instance lives in the article pane. JS / WebGL / WebRTC / plugins / DevTools / LocalStorage / IndexedDB / app cache / fullscreen are all OFF. Strict CSP (`default-src 'none'; img-src viaduct-img: data:; style-src 'unsafe-inline'`) blocks every external load. The `viaduct-img://` URI scheme routes images through our `ImageCache` so WebKit gets ZERO direct internet access. Link clicks intercepted via `decide-policy` and routed to `xdg-open`. Real-world session peak measured at **292 MB / 500 MB budget**. Adding additional WebViews (e.g. for popovers, second windows) is forbidden — typography fidelity comes from this single locked-down instance only.
5. **Error types:** public / library-surface errors use `thiserror` variants under `ViaductError`. `anyhow` is tolerated only in binary glue (`main.rs`, scratch bins). Do not add `anyhow::Result` to module APIs.
6. **No scope creep.** Don't refactor adjacent code while passing through. Don't add abstractions for hypothetical future callers. Three similar lines beats a premature helper.
7. **No new top-level dependencies without asking.** The crate list in `Cargo.toml` is deliberate. If you think you need another, propose it with the specific NNW symbol you're porting.
8. **Default to no comments.** Only write one when the *why* is non-obvious (hidden constraint, NNW-specific quirk being preserved, workaround for a real bug). Never leave "ported from X.swift:123" breadcrumbs — git blame + this file are enough.
9. **CI must stay green.** Before reporting a task complete: `cargo fmt`, `cargo clippy --all-targets -- -D warnings`, `cargo test --all`.
10. **Don't commit or push.** Ever, unless the user explicitly asks.
